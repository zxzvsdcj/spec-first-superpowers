# 项目宪法（Constitution） — OpenSpec 模式

版本：2.0
适用范围：本项目所有 AI 生成的 spec、design、tasks、代码均必须遵守此宪法。
违反任何条款均为 blocker，必须立即停止并修正。

**本宪法在每个质量闸门(G1-G4)中被主动校验。**

## 1. 核心使命

让 AI 按照工程级最佳实践工作，杜绝凭感觉编码。

最高铁律：
1. 必须先完成 SDD 全流程才允许写代码。
2. 默认 OpenSpec 模式，除非明确为全新项目且文件极少。
3. 所有规划持久化到文件，聊天记录不可作为唯一记忆。
4. UI/UX 任务必须先生成设计系统并用户确认。
5. 所有实现采用 TDD-First（先写测试再实现）。
6. 代码审查为强制步骤（两阶段：spec 符合 + 代码质量）。

## 2. 代码质量 <!-- G4 校验 -->

- Clean Code · SOLID · DRY · KISS
- 公共接口完整文档
- 命名规范统一
- 禁止魔法值
- 代码风格工具强制
- 禁止循环依赖
- 每次提交通过 linter + tests

## 3. 测试规范 <!-- G4 校验 -->

- TDD-First：先写测试再实现
- 覆盖率：关键路径 ≥ 95%，整体 ≥ 85%
- 测试独立、可重复、无副作用
- CI 测试失败 → 阻塞合并
- 禁止"先实现后补测试"

## 4. UI/UX 一致性 <!-- G3 校验 -->

- 统一设计系统（Atomic Design）
- 响应式优先 (mobile-first)
- WCAG 2.1 AA（对比度 ≥ 4.5:1、键盘导航、ARIA）
- 交互一致（按钮/导航/反馈统一）
- 国际化支持

## 5. 性能与安全基线 <!-- G4 校验 -->

- 首屏 < 2.5s (P90)
- 交互 < 100ms
- 主包 < 400KB (gzip)
- API < 200ms (P95 < 400ms)
- 禁止明文密钥/凭证
- OWASP Top 10 防护

## 6. 文档分离 <!-- G1 校验 -->

- spec.md：纯产品视角 (What & Why)，禁止技术细节
- design.md：工程视角 (How)，引用 constitution
- 违反分离 → blocker

## 7. 文件持久化 <!-- G2 校验 -->

- task_plan.md（编号检查清单 + 状态）
- findings.md（发现 + 调研）
- progress.md（日志 + 测试结果 + 验证证据）
- 每步行动前重读 task_plan.md
- 每步完成后更新 progress.md

## 8. 闸门检查点速查

| 闸门 | 校验本宪法条款 |
|------|-------------|
| G1 | §1(使命) §6(文档分离) |
| G2 | §7(文件持久化) |
| G3 | §4(UI/UX) |
| G4 | §2(代码质量) §3(测试) §5(性能安全) |

- [ ] task_plan.md 已生成并更新？

- [ ] TDD 先写了测试？
- [ ] 代码通过 linter + tests？
- [ ] 两阶段审查（spec + quality）通过？
